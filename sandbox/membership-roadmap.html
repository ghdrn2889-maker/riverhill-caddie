리버힐 캐디 앱 · 방향 전환

개인 도구 → 회원제 서비스

김홍구님 한 사람을 위해 지어진 앱을, 리버힐 캐디 전체가 각자 가입해 쓰는 서비스로 다시 세웁니다. 아래는 그 결정과 실제로 지어갈 순서입니다.

규모 · 리버힐 전체(~수백) 로그인 · 네이버 board 읽기 · 공용 전용 계정 저장소 · SQLite

지금 무엇이 바뀌나

지금 앱은 이름·부·쿠키·모든 데이터가 서버에 하나씩만 존재합니다(완전 1인용). 회원제는 이 뼈대를 바꿉니다:

  • 정체성 — 네이버 로그인으로 "누구"인지 구분. 단, 네이버 이름 ≠ board 이름일 수 있어 실명(배치표 이름)은 가입 시 따로 입력.
  • 판독 — board는 공용 리더 계정으로 한 번 읽고, 회원 이름마다 나눠 판단. 회원이 늘어도 Gemini 비용은 거의 그대로.
  • 데이터 — 오늘 상황·근무일지·카트·구독·설정을 전부 회원별로 분리 저장.
  • 개인설정 — 집 거리·차량·주유소·연비·부(部)가 회원 프로필로. 유류비는 이 프로필 위에 얹습니다.

지어갈 순서

1
뼈대 — 계정과 데이터 분리
겉모습·기능은 그대로, 속만 회원별로
  • SQLite 도입, 회원·세션·프로필 테이블 설계
  • 네이버 로그인(OAuth) 연동
  • 기존 데이터를 김홍구님 = 1번 회원으로 이관 — 지금 쓰던 게 안 끊기게
2
가입 · 온보딩
캐디가 스스로 가입하는 흐름
  • 네이버 로그인 → 실명(배치표 이름)·부·집 거리·차량번호 입력
  • 동명이인은 순번으로 구분
  • board 판독을 공용 리더 계정으로 전환
3
다부(部) 판독 + 기능 이식
1·2·3부를 모두 다루고, 개인 기능을 회원별로
  • 지금 3부 전용인 판독(티오프 하한·N팀·확정선)을 부별로 일반화
  • 근무 판단·근무일지·카트 점검을 회원별로
  • 유류비 자동화(오피넷) — 회원마다 자기 주유소·연비. API 키는 서버에 하나
4
운영 · 보안 · 개인정보
수백 명이 믿고 쓰려면
  • 실서버로 이전(집 PC는 꺼지면 전원 먹통)
  • 개인정보 동의·처리방침, 회원별 접근 차단, 백업·모니터링

먼저 처리해야 할 현실 작업

🔧 이건 계정·권한 문제라 김홍구님(또는 카페 관리자)만 할 수 있습니다
  1. 네이버 개발자센터 앱 등록김홍구내가 화면 안내 — 네이버 로그인 Client ID/Secret 발급, 콜백 주소 등록.
  2. 공용 리더 네이버 계정 생성 + 리버힐 카페 가입 승인김홍구/관리자 — board 읽기 전용. 카페 관리자 승인이 필요.
  3. 호스팅 결정함께 — 수백 명이면 집 PC는 불안정. 실서버(클라우드) 이전 시점을 잡아야 함. 1단계는 지금 환경에서 진행 가능.
  4. 오피넷 무료 API 키김홍구내가 안내 — 유류비용, 3단계에서 필요.

⚠️ 개인정보 책임이 커집니다

이제 수백 명의 실명·차량번호·근무기록·사진을 보관합니다. 이건 개인정보라 가입 동의·회원별 분리·철저한 접근 차단이 필수이고, 공개된 GitHub 저장소에는 어떤 회원 데이터도 절대 올라가면 안 됩니다. 4단계에서 제대로 다루되, 1단계부터 이 원칙으로 설계합니다.

※ 유류비 기능은 지금 붙이지 않고 3단계에서 회원 프로필 위에 얹습니다 — 그래야 다시 안 뜯습니다. 이 문서는 진행하며 갱신됩니다.
