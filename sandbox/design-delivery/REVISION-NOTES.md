# 디자인 수정·결정 로그

> 사용자와 디자인 AI가 확정한 내용을 구현 AI에게 전달하기 위한 누적 문서다.

## 근무 시스템 이해 — `WORK-SYSTEM.md` 반영

- 앱의 핵심 목적은 스페어 캐디가 겪는 `오늘 근무하는가, 언제 확정되는가`의 대기 불확실성을 줄이는 것이다.
- 김홍구 사용자는 3부 캐디이며 다른 부의 소식은 현재 상태에 영향을 주지 않는다.
- 회색 이름칸은 스페어 대기, 흰색 이름칸은 근무 확정이다.
- 순번은 사용자의 고정 기준점이지만 순번 숫자만으로 근접도를 판단할 수 없다.
- 실제 근접도는 `현재 커트라인 순번 ↔ 내 순번`의 차이가 확인될 때만 계산한다.
- 커트라인이 발표되지 않았다면 `앞으로 N명`, 진행률, 근접도를 표시하지 않는다.
- 커트라인이 발표됐다면 현재 확정 범위, 남은 순번 수, 근접도 변화를 시각화한다.
- 대기열은 추가·취소·노쇼·순번 교환으로 바뀌거나 번복될 수 있으므로 마지막 갱신 시각과 변경 강조가 중요하다.
- 스페어는 휴무가 아니며 당일 근무로 바뀔 수 있는 활성 대기 상태다.
- 확정되면 스페어 화면에서 행동 진행 보드로 전환한다.
- 사용자 최신 설명에 따라 최종 UI 계약은 `근무 확정 = 티오프 시각·코스 매칭 완료`로 유지한다. 원격 인계서의 `흰색이지만 티오프가 아직 안 보일 수 있음` 문장과 충돌할 경우 사용자의 최신 설명을 우선한다.
- 스페어 대시보드는 최소 두 데이터 상황을 별도로 설계해야 한다.
  1. 커트라인 미발표: 내 순번 + 감시 정상 + 다음 안내
  2. 커트라인 발표: 현재 커트라인 + 앞으로 남은 순번 + 근접도

## 검토 중 — 스페어 대시보드 재설계

- 기존 순번 큐 `2·3·4·5·나`는 확인되지 않은 앞 순서를 사실처럼 보이게 할 수 있어 최종안으로 확정하지 않았다.
- 정확한 데이터는 `3부 스페어`, `myPosition`, 명시된 커트라인, 감시 상태뿐이다.
- 새 비교 시안 3개를 `spare-explorations.html`에 제작했다.
  - S1 신호 관제판: 순번과 감시 신뢰 중심
  - S2 스페어 대기표: 확인된 값과 미확인 값의 구분 중심
  - S3 라이브 감시 흐름: 최근 확인과 다음 자동 행동 중심
- 세 시안 중 최종 선택 전이므로 구현 AI는 아직 기존 스페어 화면을 교체하지 않는다.
- `WORK-SYSTEM.md`를 반영한 재설계 3안을 `spare-explorations-v2.html`에 추가했다.
- V2 시안은 각 디자인에서 커트라인 미발표·3번 확정·5번 확정 상태를 전환해 검토할 수 있다.
- 가독성과 즉시 이해를 더 강화한 3차 후보를 `spare-dashboard-finalists.html`에 제작했다.
- 3차 후보는 장식적 계기판을 제거하고 `현재 커트라인 / 내 순번 / 남은 인원`의 직접 비교를 최우선으로 한다.

## 2026-07-18

### 다크모드 연기

- 메인 라이트 화면과 상태 설계를 먼저 완성한다.
- 다크모드는 이번 구현 범위에서 제외하고 추후 별도 작업으로 검토한다.
- 구현 AI는 이번 인계물을 이식하면서 다크모드를 임의로 추가하지 않는다.
- `index.html`은 라이트 테마로 고정했고 `states.html`은 다크모드 CSS 없이 다시 작성했다.

### 근무 확정과 티오프의 관계 수정

- **근무 확정은 티오프 배정이 이미 매칭된 상태**다.
- 근무 확정이면 티오프 시각과 코스가 반드시 함께 잡힌다.
- `근무 확정 + 티오프 아직 없음`은 실제 업무 상태가 아니므로 삭제했다.
- 상태 레퍼런스를 다음 6개로 수정했다.

  1. 근무 확정 + 티오프 있음
  2. 스페어 대기
  3. 지금 차례
  4. 휴무
  5. 정보 없음
  6. 로딩 / 오류

### 구현 AI가 갱신해야 할 원격 계약

`FRONTEND-CONTRACT.md`와 기존 인계서에서 `근무 확정 + 티오프 아직 없음`을 필수 상태로 적은 부분을 삭제하고 아래 업무 규칙을 반영한다.

> 근무 확정 상태는 티오프 시각·코스 매칭 완료를 의미한다. 근무 확정인데 티오프가 없는 별도 UI 상태는 만들지 않는다.
