# 디자인 수정·결정 로그

> 사용자와 디자인 AI가 확정한 내용을 구현 AI에게 전달하기 위한 누적 문서다.

## 2026-07-21 — 상태 구성 수정

### `정보 없음` 대시보드 제외

- 오늘 팀 예약이 종료되면 바로 다음 날 배치표가 공지되는 운영 흐름이므로 별도의 `정보 없음` 대시보드는 필요하지 않다.
- 기존 상태 레퍼런스의 `정보 없음/empty` 화면은 최종 상태 구성에서 제외한다.
- API가 일시적으로 비어 있거나 데이터를 읽지 못한 경우에는 `정보 없음`으로 해석하지 않고 로딩·오류 상태로 처리한다.
- 구현 AI는 `empty` 응답의 실제 발생 조건을 점검하고, 정상 업무 상태처럼 별도 대시보드를 만들지 않는다.

### 스페어 불확실성 단계 설계 보류

- 안정 대기·준비 대기·곧 내 차례처럼 스페어 대응 단계를 나누는 방안은 유용하지만, 실제 업무 규칙과 알림 가능 범위를 시스템 구현 AI와 추가 논의해야 한다.
- 협의 전까지 근접도나 행동 가능 시간을 추가로 추정하지 않는다.

## 근무 시스템 이해 — `WORK-SYSTEM.md` 반영

- 앱의 핵심 목적은 스페어 캐디가 겪는 `오늘 근무하는가, 언제 확정되는가`의 대기 불확실성을 줄이는 것이다.
- 김홍구 사용자는 3부 캐디이며 다른 부의 소식은 현재 상태에 영향을 주지 않는다.
- 회색 이름칸은 스페어 대기, 흰색 이름칸은 근무 확정이다.
- 순번은 사용자의 고정 기준점이지만 순번 숫자만으로 근접도를 판단할 수 없다.
- 실제 근접도는 `현재 커트라인 순번 ↔ 내 순번`의 차이가 확인될 때만 계산한다.
- 커트라인이 발표되지 않았다면 `앞으로 N명`, 진행률, 근접도를 표시하지 않는다.
- 커트라인이 발표됐다면 현재 확정 범위, 남은 순번 수, 근접도 변화를 시각화한다.
- 대기열은 추가·취소·노쇼·순번 교환으로 바뀌거나 번복될 수 있으므로 마지막 갱신 시각과 변경 강조가 중요하다.
- 스페어는 휴무가 아니며 당일 근무로 바뀔 수 있는 활성 대기 상태다.
- 확정되면 스페어 화면에서 행동 진행 보드로 전환한다.
- 사용자 최신 설명에 따라 최종 UI 계약은 `근무 확정 = 티오프 시각·코스 매칭 완료`로 유지한다. 원격 인계서의 `흰색이지만 티오프가 아직 안 보일 수 있음` 문장과 충돌할 경우 사용자의 최신 설명을 우선한다.
- 스페어 대시보드는 최소 두 데이터 상황을 별도로 설계해야 한다.
  1. 커트라인 미발표: 내 순번 + 감시 정상 + 다음 안내
  2. 커트라인 발표: 현재 커트라인 + 앞으로 남은 순번 + 근접도

## 검토 중 — 스페어 대시보드 재설계

- 기존 순번 큐 `2·3·4·5·나`는 확인되지 않은 앞 순서를 사실처럼 보이게 할 수 있어 최종안으로 확정하지 않았다.
- 정확한 데이터는 `3부 스페어`, `myPosition`, 명시된 커트라인, 감시 상태뿐이다.
- 새 비교 시안 3개를 `spare-explorations.html`에 제작했다.
  - S1 신호 관제판: 순번과 감시 신뢰 중심
  - S2 스페어 대기표: 확인된 값과 미확인 값의 구분 중심
  - S3 라이브 감시 흐름: 최근 확인과 다음 자동 행동 중심
- 세 시안 중 최종 선택 전이므로 구현 AI는 아직 기존 스페어 화면을 교체하지 않는다.
- `WORK-SYSTEM.md`를 반영한 재설계 3안을 `spare-explorations-v2.html`에 추가했다.
- V2 시안은 각 디자인에서 커트라인 미발표·3번 확정·5번 확정 상태를 전환해 검토할 수 있다.

### 최종 선택 — R1 근접도 계기판

- 사용자가 스페어 대시보드 최종 방향으로 `R1 근접도 계기판`을 선택했다.
- 커트라인 미발표 시 근접도와 남은 인원을 숨기고 감시 상태만 표시한다.
- 커트라인 발표 시 `현재 커트라인`, `내 순번`, `앞으로 남은 인원`을 함께 표시한다.
- 구현 AI는 기존 임의 순번 큐 대신 R1의 계산 가능한 근접도 구조를 이식한다.
- 계기판 눈금에는 `커트라인 다음 순번부터 사용자까지` 실제 대기 순서를 표시한다.
- 사용자는 계기판의 최댓값 위치에 이름으로 표시한다.
- 사용자 포함 최대 6명까지만 표시한다. 6명을 넘으면 사용자와 가장 가까운 앞 순번 5개 + 사용자를 표시한다.
- 남은 사람이 6명보다 적으면 실제 남은 인원만큼만 눈금을 표시한다.
- 표시 시작 순번 공식: `max(커트라인 + 1, 사용자 순번 - 5)`.

## 검토 중 — 지금 차례 화면

- `your-turn-explorations.html`에 지금 차례 화면 3안을 제작했다.
  - T1 긴급 신호판: 즉각적인 인지 중심
  - T2 상태 전환 보드: 스페어→지금 차례→행동 보드 흐름 중심
  - T3 확정 근거 브리핑: 커트라인·출처·감지 시각 중심
- 최종 선택 전이므로 구현 AI는 아직 `your_turn` 화면을 교체하지 않는다.

### 최종 선택 — T1 긴급 신호판

- 사용자가 `지금 차례` 화면 최종 방향으로 `T1 긴급 신호판`을 선택했다.
- 커트라인이 내 순번에 도달한 사실을 빨강 대형 히어로로 즉시 전달한다.
- 확인되지 않은 티오프·코스·출발 시각은 표시하지 않는다.
- 판정 근거와 감지 시각은 히어로 아래에서 확인할 수 있게 유지한다.
- 확정 배치가 들어오면 행동 진행 보드로 자동 전환한다.
- 구현 AI는 기존 `your_turn` 화면 대신 T1 구조를 이식한다.

## 검토 중 — 휴무 화면

- `off-explorations.html`에 휴무 대시보드 3안을 제작했다.
  - O1 그린 휴식 카드: 심리적 휴식과 상태 구분 중심
  - O2 오늘 요약 보드: 필요한 행동 0개를 명확히 표현
  - O3 조용한 자동 감시: 휴무 중에도 다음 일정 감시가 계속됨을 강조
- 최종 선택 전이므로 구현 AI는 아직 `off` 화면을 교체하지 않는다.

### O1 일러스트 재검토

- 사용자는 O1 그린 휴식 카드의 전체 방향은 선호하지만 기존 추상 골프장 일러스트는 채택하지 않았다.
- 휴식을 실제로 취하는 장면을 담은 추가 시안 3개를 `off-rest-illustrations.html`에 제작했다.
  - I1 페어웨이 해먹
  - I2 벤치에 내려놓은 캐디백
  - I3 창가의 느린 아침
- 일러스트 선택 전이므로 O1은 아직 최종 확정 상태가 아니다.
- 기존 CSS 도형 기반 I1~I3는 오브젝트 비례·정렬·원근의 완성도가 부족하다는 사용자 평가에 따라 폐기한다.
- 고품질 단일 SVG 시안을 `off-rest-premium-svg.html`에 새로 제작했다.
- 새 시안은 하나의 좌표계, 일관된 외곽선, 테라스에서 쉬는 사람과 내려놓은 캐디백 장면을 사용한다.
- 사용자 승인 전까지 휴무 최종안으로 확정하지 않는다.

### 휴무 일러스트 방식 폐기

- 사용자는 휴무 카드의 기본 방향은 선호하지만 CSS 및 SVG 일러스트 결과물의 조형 완성도가 부족하다고 판단했다.
- `off-rest-illustrations.html`과 `off-rest-premium-svg.html`의 일러스트 방식은 모두 최종안에서 제외한다.
- 휴무 화면은 별도의 장면 일러스트 없이 타이포그래피, 여백, 상태 요약과 감시 신뢰 정보로 완성하는 방향으로 전환한다.
- 구현 AI는 폐기된 일러스트 자산이나 CSS를 실제 앱에 이식하지 않는다.

### 휴무 대시보드 V2 검토

- 일러스트를 완전히 제외한 휴무 대시보드 3안을 `off-dashboard-v2.html`에 제작했다.
  - Q1 여백의 휴무: 큰 타이포그래피와 빈 공간 중심
  - Q2 행동 0 보드: 오늘 필요한 행동이 없음을 숫자로 표현
  - Q3 휴무 확인서: 상태 확인 근거와 갱신 시각 중심
- 최종 선택 전이므로 구현 AI는 아직 `off` 화면을 교체하지 않는다.
- 가독성과 즉시 이해를 더 강화한 3차 후보를 `spare-dashboard-finalists.html`에 제작했다.
- 3차 후보는 장식적 계기판을 제거하고 `현재 커트라인 / 내 순번 / 남은 인원`의 직접 비교를 최우선으로 한다.

## 2026-07-18

### 다크모드 연기

- 메인 라이트 화면과 상태 설계를 먼저 완성한다.
- 다크모드는 이번 구현 범위에서 제외하고 추후 별도 작업으로 검토한다.
- 구현 AI는 이번 인계물을 이식하면서 다크모드를 임의로 추가하지 않는다.
- `index.html`은 라이트 테마로 고정했고 `states.html`은 다크모드 CSS 없이 다시 작성했다.

### 근무 확정과 티오프의 관계 수정

- **근무 확정은 티오프 배정이 이미 매칭된 상태**다.
- 근무 확정이면 티오프 시각과 코스가 반드시 함께 잡힌다.
- `근무 확정 + 티오프 아직 없음`은 실제 업무 상태가 아니므로 삭제했다.
- 상태 레퍼런스를 다음 6개로 수정했다.

  1. 근무 확정 + 티오프 있음
  2. 스페어 대기
  3. 지금 차례
  4. 휴무
  5. 정보 없음
  6. 로딩 / 오류

### 구현 AI가 갱신해야 할 원격 계약

`FRONTEND-CONTRACT.md`와 기존 인계서에서 `근무 확정 + 티오프 아직 없음`을 필수 상태로 적은 부분을 삭제하고 아래 업무 규칙을 반영한다.

> 근무 확정 상태는 티오프 시각·코스 매칭 완료를 의미한다. 근무 확정인데 티오프가 없는 별도 UI 상태는 만들지 않는다.
