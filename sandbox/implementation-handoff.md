# 리버힐 캐디 일정 앱 — 구현 AI 인계서

> 수신: 기능 구현 담당 AI  
> 결정 시안: **행동 진행 보드(Hybrid 02)**  
> 기준 프로토타입: `sandbox/hybrid-prototypes.html`의 두 번째 화면  
> 미리보기: `http://127.0.0.1:4317/sandbox/hybrid-prototypes.html`  
> 작성 목적: 기존 기능을 보존하면서 확정된 UI·UX와 신규 기능을 실제 PWA에 구현한다.

---

## 1. 최종 디자인 결정

최종 방향은 최초 A안의 정보 구조와 가독성을 기반으로 하고, B안의 시간·행동 안내를 결합한 **행동 진행 보드**다.

핵심 문장은 다음과 같다.

> 사용자는 앱을 연 뒤 3초 안에 현재 상황, 다음 행동, 움직여야 할 시각, 알림 정상 여부를 이해해야 한다.

### 유지할 A안의 특징

- 모바일 야외 환경에서 읽기 쉬운 고대비 화면
- `오늘 / 소식 / 근무 기록`의 명확한 정보 구조
- 오늘 상태를 가장 먼저 보여주는 히어로 카드
- 일정 감시와 휴대폰 알림 상태를 분리한 신뢰 표시
- 최근 일정 소식을 중요도 순으로 보여주는 목록
- 기능과 숫자를 우선한 절제된 디자인

### 결합할 B안의 특징

- “몇 시에 움직여야 하는가”를 중심으로 한 행동 안내
- `일정 확인 → 출발 → 골프장 도착` 진행도
- 다음 행동까지 남은 시간
- 다음 확인 알림 시각
- 근무 종료 후 기록 흐름으로 자연스럽게 연결되는 구조

### 제외한 방향

- 장식 중심의 프리미엄·에디토리얼 UI
- 이모지에 의존한 상태 표현
- GPS를 켜기 위한 별도의 `출발했어요` 버튼
- PWA에서 보장할 수 없는 상시 백그라운드 GPS 추적
- GPS 결과가 불확실한데도 미출발이라고 단정하는 동작

## 2. 현재 저장소와 수정 원칙

### 실제 앱

- 화면: `public/index.html`
- 프론트 로직: `public/app.js`
- 서비스 워커: `public/sw.js`
- 매니페스트: `public/manifest.webmanifest`
- 서버: `src/server.mjs`
- 오늘 상태 계산: `src/today.mjs`
- 푸시: `src/push.mjs`
- 근무 기록: `src/worklog.mjs`

`docs/`는 배포용 복사본으로 보인다. 구현 시 원본과 배포본의 관계를 먼저 확인하고, 중복 파일을 수동으로 서로 다르게 수정하지 않는다.

### 디자인 자료

- 최초 분석: `design-brief.html`
- 방향 제안서: `sandbox/ui-design-directions.md`
- 최초 3안: `sandbox/prototypes.html`
- A+B 발전안: `sandbox/hybrid-prototypes.html`

### 작업 원칙

1. 기존 API와 데이터 구조를 먼저 재사용한다.
2. 기존 푸시·최근 소식·근무 기록 기능을 제거하지 않는다.
3. UI 프레임워크를 새로 도입하지 않는다.
4. 모바일 우선 PWA와 현재의 가벼운 로딩 성능을 유지한다.
5. GPS·날씨 기능 실패가 기존 일정 알림을 방해해서는 안 된다.
6. 위치와 기상 API 키 같은 민감 정보는 클라이언트에 저장하지 않는다.

## 3. 목표 화면 구조

### 하단 내비게이션

```text
오늘 | 소식 | 근무 기록
```

초기 구현에서는 SPA 라우터 없이 섹션 전환 방식으로 만들어도 된다. 브라우저 뒤로가기와 새로고침 후 선택 상태는 일관되게 처리한다.

### 오늘 화면 우선순위

```text
1. 날짜와 현재 시각
2. 오늘 내 상황
3. 행동 진행 보드
4. 일정 감시·알림 전달 상태
5. 골프장 날씨
6. 최근 일정 소식 요약
```

### 행동 진행 보드 기본 예시

```text
오늘 내 상황
3부 스페어 · 2번째

다음 행동 · 집에서 출발
13:42
40분 남음

일정 확인 ━━━ 출발 ━━━ 골프장 도착

13:32에 다시 알려드릴게요
```

시간 숫자는 가장 높은 시각적 위계를 갖는다. 진행도는 색상뿐 아니라 라벨과 현재 단계 강조로 구분한다.

## 4. 오늘 상태별 행동 보드

기존 상태값의 정확한 명칭과 계산은 `src/today.mjs`를 기준으로 확인한다. UI에서는 최소한 다음 네 상태를 표현한다.

### 근무 확정

```text
상태: 오늘 근무 확정
다음 행동: 출발 / 골프장 도착 / 티오프
주요 시간: 계산된 출발 시각 또는 도착 권장 시각
진행도: 일정 확인 → 출발 → 도착 → 티오프
```

### 턴 대기

```text
상태: 턴 대기
다음 행동: 연락 대기 또는 출발 준비
주요 정보: 현재 턴, 예상 확인 시점
```

### 스페어 대기

```text
상태: 3부 스페어 · 2번째
다음 행동: 출발 준비
주요 정보: 골프장 도착 권장 시각, 출발 예정 시각
```

### 휴무·일정 없음

```text
상태: 오늘 예정된 근무 없음
다음 행동: 없음
행동 보드: 휴식 상태로 축소
최근 소식과 다음 예정 일정은 계속 표시
```

### 로딩·데이터 오류

- 로딩 중에는 이전 정상 데이터를 가능하면 유지하고 갱신 상태를 표시한다.
- 상태 API 오류를 휴무로 표현하지 않는다.
- `일정을 확인하지 못했어요`와 재시도 동작을 제공한다.

## 5. 출발 시각 계산

출발 시각은 다음 값을 기반으로 계산한다.

```text
출발 시각 = 골프장 도착 권장 시각 - 예상 이동시간 - 준비 여유시간
```

초기 버전에서는 사용자 설정값을 사용한다.

- 집→골프장 기본 이동시간
- 도착 목표 여유시간
- 알림 사전 시간: 기본 10분

지도·교통 API가 없는 동안 이동시간을 실시간 교통 정보처럼 표현하지 않는다. 고정 설정값이면 `설정한 이동시간 기준`이라고 명시한다.

### 시간 상태

```text
출발 30분 이상 전   → 준비 단계
출발 10분 전        → 곧 출발
출발 시각~10분 후   → 출발 확인 필요
출발 10분 이상 지남 → 지연 가능성 강조
도착 반경 진입      → 도착
```

시간 계산은 브라우저와 서버 모두 `Asia/Seoul` 기준으로 일관되게 처리한다.

## 6. 출발 확인 알림

GPS를 쓰지 않아도 서버 시계만으로 동작해야 하는 핵심 안전장치다.

### 1차 알림

출발 예정 10분 전에 전송한다.

```text
곧 출발할 시간입니다
14:20까지 도착하려면 13:42에 출발하세요.
```

### 2차 알림

다음 조건에서 출발 예정 10분 후 전송한다.

- 첫 알림을 누르지 않음
- GPS 판정이 `미출발` 또는 `불확실`
- 앱에서 출발 상태가 확인되지 않음

```text
출발 여부를 확인해주세요
예정 출발 시각이 10분 지났습니다.
```

### 알림 클릭 처리

서비스 워커의 `notificationclick`에서 오늘 화면을 열거나 기존 창에 포커스를 준다. URL 파라미터 또는 내부 상태로 `출발 확인 진입`임을 전달한다.

```text
알림 클릭
→ 오늘 화면 열기
→ 위치 권한이 이미 허용되었다면 GPS 판정 자동 시작
→ 행동 진행 보드를 판정 결과로 갱신
```

알림 클릭 사실과 GPS 판정 결과를 서버에 전달하면 불필요한 2차 알림을 취소할 수 있다. 단, 기존 푸시 기능을 깨지 않도록 별도 이벤트/API로 추가한다.

## 7. GPS 기능 범위

GPS는 상시 추적기가 아니라 **알림을 눌러 앱이 열린 시점의 출발 판정 보조 기능**이다.

### 권한 UX

- 최초 한 번만 위치 권한의 이유를 설명한다.
- 브라우저 권한 요청 전에 자체 안내 문구를 보여준다.
- 권한 거부 시 앱의 일정·알림 기능은 그대로 작동한다.
- 이미 권한이 `granted`라면 알림 클릭 후 별도 버튼 없이 위치 확인을 시작한다.

권장 설명:

```text
출발 여부와 도착 시간을 자동으로 확인하려면
위치 사용을 한 번만 허용해주세요.

위치는 출근 시간대에 앱이 열렸을 때만 확인합니다.
```

### 판정 입력

- 현재 위치와 집 기준점의 거리
- 연속 위치 표본 3개 이상
- 각 표본의 `accuracy`
- 일정 시간 동안 실제 변위
- `speed` 값이 있을 경우 보조 지표
- 골프장까지 거리의 감소 여부

`speed` 하나만으로 차량 이동을 판정하지 않는다.

### 초기 권장 판정값

실기기 테스트 후 조정할 임시값이다.

```text
집 기준점에서 300m 이상 떨어짐
→ 출발 가능성 높음

집 반경 150m 이내
+ GPS 정확도 100m 이하
+ 유의미한 변위 없음
→ 아직 출발 전 가능성 높음

2분 동안 200m 이상 이동
+ 골프장까지 거리 감소
→ 이동 중

GPS 정확도 100m 초과 또는 표본 부족
→ 확인 중 / 판단 불가
```

### UI 표현

#### 이동 중

```text
골프장으로 이동 중
현재 위치 기준 예상 도착 14:18
```

#### 아직 출발 전

```text
아직 집 근처에 있는 것으로 보여요
출발 예정 시각은 13:42입니다.
```

#### 판단 불가

```text
위치가 정확하지 않아 출발 여부를 확인하지 못했어요
출발 예정 시각은 13:42입니다.
```

### 저장·개인정보 원칙

- 가능하면 집 좌표와 원시 GPS 좌표는 기기 로컬에만 저장한다.
- 서버에는 `departed / not_departed / uncertain`과 판정 시각만 전달한다.
- 원시 위치를 서버에 전송해야 한다면 저장 목적, 보유 기간, 삭제 방법을 먼저 정의한다.
- 위치 권한을 끄거나 저장된 기준 위치를 삭제할 수 있는 설정을 제공한다.

### 명시적 한계

- PWA가 닫힌 상태에서 지속 위치 추적을 보장하지 않는다.
- 서비스 워커에서 GPS를 조회하려고 구현하지 않는다.
- 백그라운드 GPS 실패를 출발 실패로 간주하지 않는다.
- 완전 자동 백그라운드 이동 감지가 필수가 되면 Android 네이티브 앱을 별도 검토한다.

## 8. 기상청 날씨 기능

사용자 위치가 아니라 리버힐 골프장 고정 좌표의 날씨를 조회한다.

### 데이터

- 현재 기온
- 하늘 상태
- 현재 강수와 강수량
- 앞으로 1~6시간 강수 가능성
- 풍속
- 필요 시 기상특보

### 권장 UI

행동 보드 아래에 작은 현장 정보 카드로 둔다.

```text
골프장 날씨
현재 흐림 27°C · 남서풍 4m/s
15시부터 약한 비 가능성
우의 준비 권장
```

날씨는 행동 문구로 요약한다.

- 강수 예상 → `우의 준비 권장`
- 강풍 → `강한 바람에 유의하세요`
- 폭염 → `물과 냉감용품을 준비하세요`
- 특보 없음 → 불필요한 경고를 만들지 않음

### 구현 구조

```text
기상청 API
→ 서버에서 조회 및 캐시
→ 앱 전용 weather API
→ 클라이언트 표시
```

- API 인증키를 클라이언트에 넣지 않는다.
- 서버에서 10~20분 캐시한다.
- 기상청 장애 시 마지막 정상 데이터와 갱신 시각을 보여주거나 카드를 숨긴다.
- 날씨 API 장애가 오늘 일정 API 응답을 실패시키지 않게 분리한다.
- 날씨는 참고 정보이며 근무 취소·출근 여부를 자동 결정하지 않는다.

## 9. 알림 신뢰 상태

현재의 단일 상태 문구를 다음 두 상태로 분리한다.

### 일정 감시

서버가 게시판을 정상적으로 확인하고 있는지 표현한다.

```text
정상: 일정 감시 정상 · 2분 전 확인
지연: 일정 확인이 늦어지고 있어요
오류: 일정을 확인하지 못하고 있어요
```

### 알림 전달

현재 기기의 푸시 구독이 유효한지 표현한다.

```text
정상: 이 폰 알림 정상
권한 없음: 알림 권한이 꺼져 있어요
구독 없음: 이 폰이 알림에 연결되지 않았어요
구독 오류: 알림 연결을 복구해주세요
```

정상일 때는 작은 초록 상태줄로 조용히 표현하고, 오류가 있을 때만 복구 행동을 강조한다.

## 10. 근무·세무 기록

기존 기능을 보존하되 첫 화면에서 분리한다.

### 목표 구조

- 하단 `근무 기록`에서 별도 화면 또는 섹션으로 진입
- 이번 달 근무일·주행거리·예상 유류비 요약
- 확인 대기 및 사진 미입력 건 우선 노출
- 날짜 선택 후 사진 3장과 계기판 입력
- CSV 및 증빙 PDF 기능 유지

초기 UI 개편에서 데이터 모델을 불필요하게 변경하지 않는다.

## 11. 접근성·테마

- 주요 터치 영역 최소 높이 48px
- 상태를 색상만으로 구분하지 않음
- 시간 숫자에 tabular number 적용
- 라이트·다크 테마 모두 제공
- 야외 역광에서 핵심 시간과 상태의 대비 확보
- `prefers-reduced-motion`에서 애니메이션 축소
- 로딩, 성공, 실패를 스크린리더가 인식할 수 있도록 상태 영역 사용
- 하단 내비게이션과 현재 탭에 적절한 ARIA 적용

## 13. 완료 기준

### UI

- 앱 실행 후 오늘 상태와 다음 행동을 3초 안에 파악할 수 있다.
- 네 가지 오늘 상태와 오류 상태가 모두 올바르게 표현된다.
- 360px 폭 화면에서 가로 스크롤이 생기지 않는다.
- 기존 최근 소식, 알림, 근무 기록, CSV/PDF 기능이 유지된다.

### 알림

- 출발 시각이 없는 날에는 출발 알림을 만들지 않는다.
- 같은 일정에 중복 알림이 전송되지 않는다.
- 시간 변경 시 기존 예약을 취소하고 새 시각으로 갱신한다.
- 알림 클릭 시 오늘 화면과 해당 행동 상태가 열린다.

### GPS

- 권한 거부·위치 오류가 기존 앱 기능을 막지 않는다.
- 불확실한 좌표를 미출발로 단정하지 않는다.
- 원시 위치 데이터의 저장 여부가 코드와 UI에서 명확하다.
- 앱이 백그라운드에서 계속 추적된다고 사용자에게 오해를 주지 않는다.

### 날씨

- 기상청 키가 클라이언트 번들·HTML에 노출되지 않는다.
- API 장애 시 일정 화면은 정상적으로 열린다.
- 관측·예보 기준 시각 또는 갱신 시각을 표시한다.

## 14. 이번 디자인 작업에서 실제로 완료된 것

다음 항목은 **샌드박스에서 완료된 설계·프로토타입**이다.

- 현재 UI·UX 문제 분석
- 서로 다른 최초 디자인 3안 제작
- A안과 B안 비교 및 방향 확정
- A 기반 하이브리드 2안 제작
- 행동 진행 보드를 최종 시안으로 결정
- 출발 전·후 알림 흐름 설계
- 알림 클릭 후 GPS 판정 흐름 설계
- GPS의 PWA 한계와 판정 안전장치 정의
- 기상청 날씨 카드와 서버 연동 구조 정의
- 알림 신뢰 상태를 감시/전달로 분리

다음 항목은 **아직 실제 앱에 구현되지 않았다**.

- `public/` 실제 UI 변경
- 출발 시각 계산 API
- 출발 전·후 푸시 예약
- 알림 클릭 이벤트 기록
- GPS 권한 및 위치 판정 코드
- 기상청 API 연동
- 하단 내비게이션
- 다크모드

구현 담당 AI는 프로토타입을 그대로 복사하기보다 이 문서의 상태·오류·접근성·데이터 규칙까지 포함하여 실제 앱으로 옮겨야 한다.
