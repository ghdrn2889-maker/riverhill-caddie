## 1. 개요 — 뭘·누가·왜

- **사용자:** 김홍구. 리버힐 골프장 **3부 캐디**(주로 스페어/대기, 근무 확정 시 티오프 배정).
- **문제:** 캐디의 하루는 "오늘 내가 나가나? 몇 시에? 언제 집에서 출발?"의 **대기 불확실성**. 이를 확인하려 네이버 카페·카톡을 끝없이 새로고침.
- **앱의 임무:** 카페·카톡을 **대신 감시**해, 내 순번·근무·티오프가 정해지는 순간 **폰으로 먼저 알림**. + 덤으로 근무일 자동 기록(세무 증빙).
- **플랫폼:** PWA(안드로이드 크롬), 폰 세로 전용, 한국어, **단일 사용자**.

## 2. 현재 상태 (2026-07-17)

**기능:** 안정화 단계. 카페·카톡 감지 → 판독 → 푸시 파이프라인 작동.
**UI:** 확정 시안(행동 진행 보드)으로 Phase 1 구현 완료. 하단 내비(오늘/소식/근무기록)·알림 신뢰 상태 분리·근무기록 보존.

**최근 해결된 것:**
- 알림 미도달(죽은 구독) → 발송 결과 로깅 + 구독 자가복구로 해결.
- 카톡 감지(방필터 제거·커트라인 인식), 배치표 최신 감지.
- **배경색=근무/스페어 판정 최우선**(흰색=근무확정, 티오프 없어도 유지). 스페어 오판 수정.
- **배치표 티오프 정확 판독**: `GEMINI_BOARD_MODEL=gemini-flash-latest`(배치표 이미지만). flash-lite는 조밀한 표 오독 → flash-latest는 정확(무료 등급, 배치표만 써서 한도 내).

**미결/다음:**
- **디자인 보강** — 매일 보는 "스페어 대기" 화면 미설계 등 → [🎨 디자인 AI 인계서](DESIGN-TASK.md)로 디자인 AI에게 제작 의뢰(진행 중).
- UI Phase 2~4(출발 알림·날씨·GPS) — [로드맵](#6-로드맵) 참고. 실사용 경과 관찰 중.

## 3. 아키텍처 지도

```
[네이버 카페 API]  [카톡→MacroDroid→POST /api/ingest]
        \                    /
         → src/crawler.mjs (폴링·heartbeat·댓글감지)
                 ↓
         src/judge.mjs  ← 두뇌. 글+이미지를 Gemini로 판단→verdict
                 ↓         (배치표는 flash-latest, 그 외 flash-lite)
         src/today.mjs  ← 오늘 상황판(running state). 번복 감지.
                 ↓
         src/server.mjs ← API·정적서빙·푸시 라우팅
                 ↓
         src/push.mjs (web-push/FCM) → 폰
         src/worklog.mjs ← 근무일지·세무 증빙

[프론트] public/index.html(화면·CSS) + app.js(데이터 배선) + sw.js(서비스워커)
```

- **판독 원칙:** "모델은 위치·색만 읽고, 산수는 코드가." teeGrid(표 전사)→코드가 순번으로 티오프 확정.
- **피드-우선:** 관련/무관 모두 `recent.json`에 저장 → 놓침 구조적 불가.
- 파일별 상세: [데이터 계약](CONTRACTS.md).

## 4. 역할 & 핸드오프 규약 (드리프트 방지 — 매우 중요)

| 담당 | 소유 | 안 건드림 |
|---|---|---|
| **디자인 AI** | `public/index.html`(화면·CSS), 상태별 정적 레퍼런스 | 로직·서버 |
| **구현 AI(Claude Code)** | `public/app.js`·`src/*`·배포 | **CSS/디자인** |
| **사용자(김홍구)** | 방향 결정·검수·AI 간 전달 | — |

- **접착제 = [데이터 계약](CONTRACTS.md).** 디자인 AI는 계약의 요소 ID·상태 목록에 맞춰 화면을 만들고, 구현 AI는 그 화면에 데이터만 배선한다.
- **핵심 교훈:** 넘겨받은 디자인은 **그대로 이식**한다(재해석 금지). 바꿔야 할 이유가 있으면 사용자에게 말하고 결정한다. → 상세 [프론트엔드 계약](FRONTEND-CONTRACT.md).
- **결정이 생기면** [결정 로그](DECISIONS.md)에 한 줄 남긴다(무엇을·왜). 재논쟁 방지.
- **세션 끝에** 구현 AI가 이 허브의 "현재 상태"·"로드맵"을 갱신한다.

## 8. 용어집

- **부(部):** 1·2·3부. 각 부 독립. 김홍구=3부. 3부 티오프는 **16시 이후**.
- **스페어:** 대기(당일 근무로 바뀔 수 있음). ≠ 휴무.
- **근무 확정:** 배치표 이름칸 **흰색**(회색=스페어). 근무 확정이면 임시 티오프가 매칭됨.
- **티오프:** 경기 시작 시각(HH:MM). OUT/IN=코스. 도착=티오프−준비, 집출발=도착−이동.
- **커트라인:** "○○님까지 근무" — 그 사람(포함)까지 순번 근무 확정.
- **teeGrid:** "OUT n부 IN" 티오프표를 {순번,시간,코스} 배열로 전사한 것(모델 눈대중 방지).
- **번복(reversal):** 스페어↔근무, 티오프 변경 등 이전 예측이 뒤집힘 → 강조 알림.
